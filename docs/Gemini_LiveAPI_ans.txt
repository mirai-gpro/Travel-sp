Gemini Live API: ブラウザからの音声ストリーミングにおけるマイク制御のベストプラクティス

背景

Gemini Live API を使ったWebアプリ（TypeScript）を開発しています。ブラウザからWebSocket経由で音声をリアルタイム送信し、Geminiが応答音声（PCM 24kHz）を返すアーキテクチャです。

バックエンド構成:

Python サーバーがWebSocket中継

ブラウザ → {type: 'audio', data: base64} → サーバー → Gemini Live API

Gemini → 応答 → サーバー → {type: 'audio', data: base64_pcm} + {type: 'transcription'} → ブラウザ

フロントエンドの現在の実装:

AudioWorkletNode でマイク音声をキャプチャ → PCM 16bit 16kHz にダウンサンプリング → base64 → WebSocket送信

クライアント側VAD（AnalyserNode + getByteFrequencyData）で無音検知 → 一定時間後に自動停止

AI応答音声（PCM 24kHz）は HTMLAudioElement で WAV blob として再生

半二重制御: AI音声再生中はマイクを停止（audioManager.stopStreaming()）

現在発生している問題

マイク開始/停止の制御が不安定: startStreaming() → AudioWorkletNode 作成 → stopStreaming() → 切断、を繰り返すとAudioContextの状態が不整合になることがある

iOS (iPhone 16/17): globalAudioContext を維持して MediaStream トラックを停止せずに再利用しているが、Audio Session が PlayAndRecord のままになり、HTMLAudioElement での再生音声が受話口から出力される

VADのタイミング: クライアント側VADで無音検知して stopStreaming() を呼んでいるが、Gemini側のVAD（turn_complete）との二重制御になっている可能性がある

AudioWorkletNode の再登録: startStreaming() のたびに audioWorklet.addModule() でプロセッサーを再登録しているが、これは正しいパターンか？

Gemini の回答
Gemini Live APIを用いたリアルタイム音声アプリケーションの開発、お疲れ様です。Web Audio API、WebSocket、そしてiOS Safari特有の挙動が絡むため、状態管理が非常に複雑になりやすい領域ですね。

現状発生している「不安定なマイク制御」「iOSの受話口問題」「VADの競合」は、「状態の頻繁な破棄と再作成」 と 「入力（Web Audio API）と出力（HTMLAudioElement）の分離」 が主な原因と考えられます。

ご質問いただいた内容に対するベストプラクティスと、より軽量で安定した実装パターンを解説します。

1. AudioContext と MediaStream のライフサイクル
結論：毎回破棄・再作成せず、セッション全体で「1つ」を維持し、フラグ制御に切り替えるべきです。

AudioContext: アプリケーション（または通話セッション）内で シングルトン（1つ） として使い回すのが Web Audio API のベストプラクティスです。作成と破棄を繰り返すと、ブラウザのハードウェアリソース解放が追いつかず、状態の不整合やメモリリークを引き起こします。

AudioWorkletNode: addModule は同一 AudioContext に対して1回だけ実行します。ノードの生成も最初のみ行い、以降は繋いだままにします。

MediaStream (getUserMedia): 毎回取得すると、そのたびにハードウェアの初期化が走り、iOS/Androidではマイクのインジケーターが点滅したり、取得に数秒の遅延が生じたりします。取得した MediaStream はセッション終了まで保持すべきです。

2. 半二重制御（AI応答中にマイク停止）の実装
NodeやStreamを破棄するのではなく、送信処理のON/OFF（フラグ制御） で半二重を実現します。

推奨手法: AudioWorkletNode と WebSocket の間にゲートを設けます。

AIの応答が開始した時 → this.canSendAudio = false にする。（マイクのPCMデータがWebSocketに送られなくなる）

AIの応答が終了した時 → this.canSendAudio = true にする。

これだけで、マイクの接続状態を維持したまま、完全に軽量な半二重通信が実現できます。

3. VAD（音声区間検出）の役割分担
結論：会話のターン制御は「Gemini側のVAD」に任せ、クライアント側VADは「ネットワーク帯域の節約（無音カット）」のみに使用すべきです。

Gemini Live API には非常に高精度なサーバーサイドVADが備わっており、文脈やポーズ（息継ぎ）を考慮してターンの終了を検知します。

クライアント側で「3.5秒無音だから送信停止（切断）」という制御を入れると、Gemini側が「ユーザーはまだ考えている途中かも」と待っている状態と競合し、対話のテンポがおかしくなります。

推奨パターンの運用:

クライアントVADが無音を検知 → WebSocketへのデータ送信を一時的にスキップ（帯域節約）。

ユーザーが話し終わる → クライアントは無音として送信をスキップしている状態。

サーバー（Gemini）が文脈から発話終了を検知 → AIの応答が返ってくる。

AIの応答を受信 → canSendAudio = false にしてマイク入力を完全にブロック。

4. iOS Safari における Audio Session と受話口問題への対処
iOSで getUserMedia を実行すると、OSの Audio Session が PlayAndRecord (通話モード) に移行します。この状態で <audio> タグ (HTMLAudioElement) で音声を再生すると、OSが「電話中である」と判断し、スピーカーではなく受話口 (Earpiece) から音声を出力してしまうのが Safari の長年の仕様（悪名高い挙動）です。

対処法：AIの応答音声も HTMLAudioElement ではなく、Web Audio API (AudioContext) を使って再生する。

入力（マイク）で使っているのと同じ globalAudioContext を使って、受信した PCM 24kHz データを再生します。

Web Audio API 内でルーティングを完結させることで、iOS 側で「ハンズフリー通話」として認識されやすくなり、スピーカーから出力されるようになります。

5. 改善されたコードパターンの提案 (Before -> After)
現状の「毎回破棄・再作成」を、「初期化1回・フラグ制御」に書き換えたクラス設計の概要です。

TypeScript
class AudioStreamManager {
  private audioContext: AudioContext | null = null;
  private mediaStream: MediaStream | null = null;
  private audioWorkletNode: AudioWorkletNode | null = null;
  
  // フラグ制御で半二重やVADを実現
  public isAiSpeaking: boolean = false; 

  // --- 1. セッション開始時に1度だけ呼ばれる初期化 ---
  async initializeAndStart(ws: WebSocket) {
    if (this.audioContext) return; // 既に初期化済みならスキップ

    // 1. AudioContext は1つだけ作成
    this.audioContext = new AudioContext({ sampleRate: 48000 });

    // 2. getUserMedia も1回だけ取得し保持
    this.mediaStream = await navigator.mediaDevices.getUserMedia({ audio: {
      echoCancellation: true,
      noiseSuppression: true,
      autoGainControl: true
    }});

    // 3. モジュール登録は1回だけ
    await this.audioContext.audioWorklet.addModule('processor.js');

    // 4. Nodeを作成して接続
    const source = this.audioContext.createMediaStreamSource(this.mediaStream);
    this.audioWorkletNode = new AudioWorkletNode(this.audioContext, 'audio-processor');
    source.connect(this.audioWorkletNode);
    this.audioWorkletNode.connect(this.audioContext.destination); // 状況に応じて不要なら外す

    // 5. フラグによる送信制御
    this.audioWorkletNode.port.onmessage = (e) => {
      // AIが喋っている間は送信しない（半二重）
      if (this.isAiSpeaking) return;

      // ※ここで簡易的なクライアントVAD判定を行い、無音なら return するのもアリ（帯域節約）
      // const isSilent = checkSilence(e.data.audioChunk);
      // if (isSilent) return;

      // 送信
      ws.send(JSON.stringify({
        type: 'audio', 
        data: base64(e.data.audioChunk)
      }));
    };
  }

  // --- 2. AIの応答状態に応じたフラグの切り替え ---
  onAiResponseStarted() {
    this.isAiSpeaking = true; // マイクからの送信をブロック
  }

  onAiResponseEnded() {
    this.isAiSpeaking = false; // マイクからの送信を再開
  }

  // --- 3. iOS対策: 受信した音声の再生も Web Audio API で行う ---
  playAiAudio(pcmData24kHz: Float32Array) {
    if (!this.audioContext) return;

    // PCMデータを AudioBuffer に変換して再生
    const buffer = this.audioContext.createBuffer(1, pcmData24kHz.length, 24000);
    buffer.copyToChannel(pcmData24kHz, 0);

    const source = this.audioContext.createBufferSource();
    source.buffer = buffer;
    source.connect(this.audioContext.destination);
    source.start();
    
    // ※実際にはストリーミング再生のためにキューイングの仕組みが必要です
  }

  // --- 4. 完全に通話を終了する時だけ全て破棄 ---
  terminateSession() {
    if (this.mediaStream) {
      this.mediaStream.getTracks().forEach(track => track.stop());
    }
    if (this.audioContext) {
      this.audioContext.close();
    }
    // nullに戻して初期状態へ
    this.audioContext = null;
    this.mediaStream = null;
    this.audioWorkletNode = null;
  }
}
まとめ
状態の保持: AudioContext, MediaStream, AudioWorkletNode はセッション中使い回し、terminateSession が呼ばれるまで破棄しない。

半二重制御: stopStreaming() で破棄するのではなく、フラグ (isAiSpeaking) で ws.send をスキップさせるだけにする。

iOS対策: AIの音声再生に <audio> 要素（HTMLAudioElement）を使わず、Web Audio API (AudioBufferSourceNode など) に統合する。

VAD: 切断処理はクライアントで行わず、Gemini側の状態に委ねる。

このアーキテクチャに変更することで、状態の不整合がなくなり、iOSでの音声ルーティングも格段に安定するはずです。
